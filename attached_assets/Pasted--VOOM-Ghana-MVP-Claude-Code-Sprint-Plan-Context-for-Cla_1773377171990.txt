# VOOM Ghana MVP — Claude Code Sprint Plan

## Context for Claude Code

You are building the MVP for VOOM, a car parts marketplace for Ghana. The repo is at `github.com/bogolobango/voom-ghana-marketplace`. The existing codebase was generated by Manus on Replit — it’s a TypeScript full-stack app with a React (Vite) client, Express server, Drizzle ORM, and is deployed on Vercel. It currently looks like a marketplace but does not function as one. Vendors cannot actually sign up, list parts, or receive orders. Buyers cannot search real inventory or transact.

**Your job is to make this a functional marketplace in the minimum number of changes possible.** Do not redesign the UI. Do not add features beyond what’s specified. Do not refactor for elegance. Ship working functionality.

-----

## Architecture Constraints

- **Stack**: Keep the existing stack — React + Vite frontend, Express backend, Drizzle ORM, PostgreSQL (Neon or Supabase, whatever is currently configured)
- **Auth**: Phone number + OTP (no email). Ghana users don’t use email for commerce. Use a simple OTP flow — for MVP, the OTP can be a static code (1234) or console-logged, with real SMS integration (Hubtel SMS API or Arkesel) added in Phase 2
- **Payments**: Use Paystack (paystack.com) as payment aggregator. Paystack handles MTN MoMo, Telecel Cash, AirtelTigo Money, Visa, Mastercard, and Apple Pay in Ghana through a single integration. Also support “Pay on Pickup” and “Cash on Delivery” as offline payment methods. Currency is GHS (Ghana Cedis). Amounts sent to Paystack must be in pesewas (subunit × 100)
- **File uploads**: Use Cloudinary (free tier, 25GB) for part images. Do not store images locally or in the DB
- **Deployment**: Vercel (frontend) — keep existing setup. Backend may need separate hosting if currently on Replit (check)
- **Mobile-first**: Every page must work on a phone screen. Ghana users are 90%+ mobile

## API Routes

Build these Express routes. Use simple middleware for auth (check session/token). Keep it RESTful and minimal.

### Auth

```
POST /api/auth/request-otp     { phone } → sends OTP, returns { success }
POST /api/auth/verify-otp      { phone, otp } → returns { token, user }
GET  /api/auth/me               → returns current user (from token)
POST /api/auth/logout           → clears session
```

### Vendor Onboarding

```
POST /api/vendors/register     { shop_name, location, whatsapp_number, description } → creates vendor profile (user must be authenticated, sets role to 'vendor')
GET  /api/vendors/:id          → public vendor profile
GET  /api/vendors              → list all vendors (public, for "View Vendors" page)
```

### Parts (Listings)

```
POST   /api/parts              → create listing (vendor auth required)
GET    /api/parts              → search/browse parts (public)
         Query params: ?q=search&category=slug&make=id&model=id&year=2018&condition=new&min_price=&max_price=&vendor_id=&page=1&limit=20
GET    /api/parts/:id          → single part detail (public)
PUT    /api/parts/:id          → update listing (vendor auth, must own)
DELETE /api/parts/:id          → soft delete / mark unavailable (vendor auth, must own)
```

### Orders

```
POST   /api/orders             → create order (buyer auth required) { items: [{ part_id, quantity }], payment_method, delivery_address?, notes? }
                                  If payment_method is 'momo' or 'card': initialize Paystack transaction, return authorization_url
                                  If payment_method is 'pickup_cash' or 'delivery_cash': create order directly with status 'confirmed'
GET    /api/orders             → list my orders (buyer sees their orders, vendor sees orders for their parts)
GET    /api/orders/:id         → order detail
PUT    /api/orders/:id/status  → update status (vendor auth) { status }
```

### Payments (Paystack)

```
POST   /api/payments/initialize   → Initialize Paystack transaction for an order. Returns { authorization_url, reference }
GET    /api/payments/verify/:ref  → Verify payment status with Paystack after redirect/callback
POST   /api/payments/webhook      → Paystack webhook endpoint (receives charge.success events). Update order payment_status and status. Must return 200 immediately.
```

### Reference Data

```
GET /api/categories            → list all categories
GET /api/makes                 → list all vehicle makes
GET /api/models?make_id=X      → list models for a make
```

### Image Upload

```
POST /api/upload               → upload image to Cloudinary, return URL (vendor auth required, accept multipart/form-data)
```

## Critical Technical Decisions

### Auth Token Strategy

Use HTTP-only cookies with JWT for web, or simple bearer tokens stored in React state (with refresh). For MVP, a session cookie is simplest. Do NOT use localStorage for tokens.

### Price Storage

All prices stored as integers in pesewas (1 GH¢ = 100 pesewas). Display formatting: `(price / 100).toFixed(2)` with “GH¢” prefix. Example: stored as 15000 → displayed as “GH¢ 150.00”

### Image Upload Flow

1. Vendor selects image on “Add Part” form
1. Frontend sends image to POST /api/upload (multipart)
1. Server uploads to Cloudinary, returns URL
1. Frontend stores URL in form state
1. On form submit, URLs are sent as part of the part creation payload
1. Use Cloudinary transformations for thumbnails: append `/w_400,h_400,c_fill` to URL

### Search Implementation

For MVP, use PostgreSQL full-text search or simple ILIKE queries. Do NOT add Elasticsearch or Algolia. The query should search `parts.title` and `parts.description`. Filter by exact match on category, make, model. Filter by range on year and price.

### WhatsApp Integration

This is not a technical integration — it’s just deep links:

```
https://wa.me/{whatsapp_number}?text={url_encoded_message}
```

The message should include the part title and a link back to the listing. This is the most important “feature” for early traction because it maps to existing behavior.

### Cart State

Use React Context or zustand for cart state. The cart lives in memory only — it resets on page refresh. This is fine for MVP. The cart is really just a convenience wrapper around the order creation flow.

## Paystack Integration (Ghana Mobile Money + Cards)

### Setup

1. Create free account at dashboard.paystack.com
1. Complete basic onboarding (Paystack Starter plan works for unregistered businesses)
1. Get test keys from Settings → API Keys & Webhooks
1. Set environment variables: PAYSTACK_SECRET_KEY, PAYSTACK_PUBLIC_KEY
1. Set webhook URL in Paystack dashboard: `https://your-domain.com/api/payments/webhook`

### How Paystack Works in Ghana

- Buyer selects MoMo at checkout → Paystack sends USSD prompt to buyer’s phone → buyer enters PIN → payment confirmed via webhook
- Buyer selects Card → Paystack shows card form → 3D Secure → payment confirmed
- All amounts in pesewas (GH¢ 100.00 = 10000 pesewas)
- Currency code: “GHS”
- Paystack settles to your bank account (next business day)

### Provider Codes for Mobile Money

```
MTN MoMo: "mtn"
Telecel Cash (formerly Vodafone Cash): "vod"  
AirtelTigo Money: "tgo"
```

### Server-Side: Initialize Transaction

```typescript
// POST /api/payments/initialize
const response = await fetch('https://api.paystack.co/transaction/initialize', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.PAYSTACK_SECRET_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    email: buyerEmail, // required by Paystack even for MoMo
    amount: totalAmountInPesewas, // e.g. 15000 for GH¢ 150.00
    currency: 'GHS',
    reference: `voom-${orderId}-${Date.now()}`, // unique reference
    callback_url: `${process.env.APP_URL}/orders/${orderId}/confirm`,
    channels: ['mobile_money', 'card'], // show both options
    metadata: {
      order_id: orderId,
      buyer_name: buyerName,
      custom_fields: [
        { display_name: 'Order ID', variable_name: 'order_id', value: orderId },
      ],
    },
  }),
});
const data = await response.json();
// data.data.authorization_url → redirect buyer here
// data.data.reference → store in order record
// data.data.access_code → for inline popup
```

  // data.data.channel → 'mobile_money' or 'card'

### Server-Side: Webhook Handler

```typescript
// POST /api/payments/webhook
import crypto from 'crypto';

function verifyWebhookSignature(req: Request): boolean {
  const hash = crypto
    .createHmac('sha512', process.env.PAYSTACK_SECRET_KEY!)
    .update(JSON.stringify(req.body))
    .digest('hex');
  return hash === req.headers['x-paystack-signature'];
}

// In route handler:
if (!verifyWebhookSignature(req)) {
  return res.status(401).send('Invalid signature');
}
const event = req.body;
if (event.event === 'charge.success') {
  const { reference, amount, metadata } = event.data;
  const orderId = metadata.order_id;
  // Update order: payment_status = 'success', status = 'paid'
}
// ALWAYS return 200 immediately — Paystack retries on non-200
return res.status(200).send('OK');
```

### Client-Side: Paystack Inline Popup (Recommended for Mobile UX)

```html

```

```typescript
// In checkout component:
function payWithPaystack(accessCode: string, orderId: string) {
  const handler = (window as any).PaystackPop.setup({
    key: import.meta.env.VITE_PAYSTACK_PUBLIC_KEY,

    channels: ['mobile_money', 'card'],

      // buyer closed popup without completing — check status
      alert('Payment was not completed');
    },
    callback: (response: { reference: string }) => {
      // Payment completed — verify on server
      fetch(`/api/payments/verify/${response.reference}`)
        .then(res => res.json())
        .then(data => {
          if (data.verified) {
            window.location.href = `/orders/${orderId}/confirm`;
          }
        });
    },
  });
  handler.openIframe();
}
```

### Paystack Test Mode

- Test secret key starts with `sk_test_`
- Test public key starts with `pk_test_`
- For MoMo testing: use phone number `0551234987` and OTP `123456`
- For card testing: use card `4084 0840 8408 4081`, expiry `04/28`, CVV `408`
- Test transactions don’t charge real money
- Webhooks work in test mode — set your webhook URL in dashboard

### Paystack Fees (Ghana)

- Local cards: 1.95% per transaction (capped at GH¢ 100)
- Mobile money: 1.5% per transaction (no cap — important for high-value parts)
- International cards: 3.9% + GH¢ 2.00
- Your take rate should be priced ABOVE Paystack fees to maintain margin

### Important: Email Requirement

Paystack requires an email for every transaction. Most Ghana car parts buyers don’t have/use email regularly. Solutions:

- Generate a placeholder email: `buyer_{phone}@voom.gh` (easiest, recommended for MVP)
- Or ask for email optionally and generate placeholder if not provided
- The email is used for Paystack receipts — buyers can ignore them

## Competitive Context (for Claude Code’s awareness)

The only YC-backed competitor in this space (Garage Mobility, YC S22) pivoted away from auto parts entirely in 2024 due to logistics complexity and currency volatility. They were a wholesale DISTRIBUTOR (held inventory, managed logistics). VOOM is a MARKETPLACE (asset-light, never touches parts). This structural difference is critical — do not build any features that would make VOOM responsible for inventory or delivery logistics. The platform connects buyers to sellers. Period.

Other digital players in Ghana (Abossey Okai Online, Tonaton classifieds) have minimal traction and no automotive-specific features. VOOM’s differentiation is: make/model/year search taxonomy, vendor verification, integrated MoMo payments, and mobile-first UX.

-----

## Lessons from Garage Mobility’s Failure (Apply These)

1. DO NOT hold inventory or manage physical logistics — you are a marketplace
1. DO NOT expand to multiple countries before proving Ghana — they spread to US/Ghana/Nigeria too fast
1. DO build WhatsApp into every flow — their WhatsApp AI agent was their most successful feature
1. DO price everything in Cedis — avoid USD exposure that killed their margins
1. DO keep the team lean — they had 40 sales people before automating; you have Julius
1. DO instrument everything — log searches, listings, inquiries, conversions for future data moat
